# Phase 3 Spec — Parent Dashboard

## Title
Adaptive Play — Phase 3: Parent Dashboard for Progress Visibility

## Goal
Create a simple parent-facing dashboard that shows:
- what the child practiced
- where the child is improving
- where more help is needed
- what the platform will focus on next

This phase makes the learning loop visible to adults and turns the system into something a parent can understand and trust.

## Why this phase matters
The child experiences the games.
The harness tracks progress.
The dashboard explains that progress in a simple way.

For the hackathon, this adds a strong story:
- AI generates the game
- the system learns from play
- parents can actually see progress

## Product experience
### Parent flow
1. Parent opens the dashboard.
2. Parent selects a child profile.
3. Parent sees a summary of recent activity.
4. Parent can review:
   - sessions played
   - areas practiced
   - strengths
   - areas that need more work
   - recommended next focus

## Scope
### In scope
- parent login or simple protected parent view
- child summary cards
- recent sessions list
- simple progress indicators
- next recommendation card

### Out of scope
- deep clinical reports
- comparisons with other children
- advanced analytics
- messaging with teachers or therapists
- formal assessments

## Core dashboard sections
### 1. Overview card
Shows:
- child name
- total sessions played
- last played date
- current focus area

### 2. Practice breakdown
Shows:
- Reading sessions
- Memory sessions
- Attention sessions

### 3. Doing well
Shows the areas where the child has improved.

Examples:
- "Doing well in memory matching"
- "Faster response time in attention games"

### 4. Needs more practice
Shows the areas where the child is still struggling.

Examples:
- "Needs more repetition on short reading words"
- "Gets distracted during longer attention rounds"

### 5. Next recommendation
Shows what the next game should focus on.

Examples:
- "Next session: easy reading words with fewer choices"
- "Next session: shorter attention rounds with fewer distractors"

### 6. Recent sessions
A simple table or list with:
- date
- skill
- score
- short summary

## Functional requirements
### FR1 — Parent view
The platform must provide a parent-facing dashboard separate from the child gameplay screen.

### FR2 — Child overview
The dashboard must display a summary for each child, including:
- name
- sessions played
- last practiced skill
- current recommended focus

### FR3 — Progress summary
The dashboard must show plain-language summaries of:
- strengths
- areas needing support
- recent changes

### FR4 — Recent session history
The dashboard must show recent sessions with:
- date
- skill
- score or accuracy
- short interpretation

### FR5 — Recommendation display
The dashboard must show the latest recommended next step generated by the harness.

### FR6 — Simple trends
The dashboard should show lightweight trends such as:
- accuracy improving
- mistakes decreasing
- more sessions completed

### FR7 — Parent-friendly language
The dashboard must avoid technical or clinical language and use simple supportive explanations.

## Non-functional requirements
- fast dashboard load time
- simple responsive layout
- easy-to-read cards and summaries
- clear separation between child view and parent view

## Suggested UI layout
### Top row
- Child name
- Sessions played
- Last active
- Current focus

### Middle row
- Doing well
- Needs more practice
- Next recommendation

### Bottom row
- Recent sessions
- Skill breakdown

## Data sources
The dashboard reads from Phase 2 data:
- child profile
- sessions table
- event summaries
- recommendations table

## Example parent summaries
### Example 1
**Doing well:** Memory matching is improving.  
**Needs practice:** Reading short words still needs repetition.  
**Next focus:** Reading game with 4 simple words and fewer distractors.

### Example 2
**Doing well:** Attention game response time is improving.  
**Needs practice:** Accuracy drops in longer rounds.  
**Next focus:** Shorter attention rounds with simpler targets.

## Technical approach
### Dashboard service responsibilities
- fetch child summary
- fetch recent sessions
- fetch latest recommendations
- render parent-friendly view

### Recommended implementation
A simple dashboard page is enough for the hackathon, using:
- summary cards
- small list/table for sessions
- one or two lightweight charts or progress bars

## Suggested dashboard metrics
- total sessions
- sessions by skill
- latest score
- average accuracy
- most common weak area
- next recommended practice

## Success criteria
Phase 3 is successful if:
- a parent can open a dashboard
- the dashboard clearly shows what the child practiced
- the dashboard explains one strong area and one weak area
- the dashboard shows what the next session will focus on

## Demo script
1. Play two or three sessions as a child.
2. Open the parent dashboard.
3. Show the practice history.
4. Show one strength and one area needing more practice.
5. Show the system's next recommendation.

## Risks
### Risk
The dashboard becomes too detailed for a hackathon.

### Mitigation
Use summary cards and a short recent sessions list.

### Risk
The language sounds too clinical.

### Mitigation
Use simple, supportive wording.

### Risk
The dashboard does not feel connected to the game.

### Mitigation
Highlight the link between recent mistakes and next-session recommendations.
